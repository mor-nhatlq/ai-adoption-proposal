obra/superpowers — áp dụng vào dự án mới vs dự án cũ

Chi Tiết Từng Phase — Dự Án Mới vs Dự Án Cũ
1 Brainstorm & Design
BrainstormingFramework Literacy
▼
Dự án mới
  • Socratic Brainstorming Brainstorming
    AI đặt câu hỏi Socratic 9 bước: context → constraints → goals → 2-3 approaches → trade-offs. Không viết code, chỉ hiểu bài toán.
  • Tạo Design Doc GATE
    Output: design spec file. Bao gồm: goals, architecture, data flow, API contracts. Human PHẢI approve trước khi code.
  • Bắt đầu từ zero — tự do chọn architecture HUMAN
    Không ràng buộc bởi code cũ. AI đề xuất architecture tốt nhất, team chọn. Có thể dùng greenfield-first approach.
Dự án cũ
  • Scan & Hiểu Codebase Trước Framework Literacy
    AI đọc toàn bộ code, map dependencies, hiểu conventions đang dùng. Framework Literacy (#13) để AI hiểu "luật chơi" của dự án.
  • Brainstorm Trong Ràng Buộc Brainstorming
    AI brainstorm nhưng phải phù hợp với architecture hiện tại. Các approach đều cần backward compatible. Không đề xuất "làm lại từ đầu".
  • Impact Analysis trong Design Doc GATE
    Design doc PHẢI có thêm: files bị ảnh hưởng, risk of breaking changes, rollback plan. Human approve.
2 Planning (TDD-First Tasks)
Writing PlansSubagent Driving
▼
Dự án mới
  • Breakdown thành TDD Tasks Writing Plans TDD
    Mỗi task 2-5 phút: viết failing test → implement → verify → commit. Exact file paths, code blocks, test commands. Không placeholders.
  • Subagent Assignment Subagent Driving
    Phân công: Planner tạo plan → Coder implement → Tester verify → Reviewer check. Mỗi agent có role rõ ràng.
  • Plan từ design doc trực tiếp
    Plan sinh từ design doc đã approve. Không cần tìm hiểu code cũ vì chưa có code. Nhanh và rõ ràng.
Dự án cũ
  • Plan Phải Reference Code Cũ Writing Plans
    Mỗi task phải ghi rõ: file nào cần sửa, function nào bị ảnh hưởng, test nào cần update. AI đọc code trước khi tạo plan.
  • Test Coverage Tasks TRƯỚC TDD GATE
    Bước bắt buộc cho dự án cũ: tasks đầu tiên là viết test cho code chưa có test. Tạo safety net trước khi thay đổi bất kỳ code nào.
  • Incremental Plan Subagent Driving
    Plan chia nhỏ hơn bình thường. Mỗi batch chỉ thay đổi 1-2 files. Verify sau mỗi batch. Không làm nhiều cùng lúc.
3 Execution (Coding)
Git WorktreesExecuting PlansParallel Dispatch
▼
Dự án mới
  • Git Worktrees cho Parallel Work Git Worktrees Parallel Dispatch
    Nhiều agents code song song, mỗi agent trong 1 worktree riêng. Backend/Frontend/DB đồng thời. Không conflict.
  • Execute Plan từng task Executing Plans TDD
    Mỗi task: viết failing test trước → implement code → test pass → commit. Không skip bước test. Không commit code chưa test.
  • Tốc độ cao từ sprint 1 Parallel Dispatch
    Không có code cũ cần tìm hiểu. AI generate code trực tiếp từ spec. Parallel dispatch (#9) cho max throughput.
Dự án cũ
  • Worktree để Isolate Changes Git Worktrees
    KHÔNG code trực tiếp trên main. Dùng worktree để isolate. Nếu fail thì discard, không ảnh hưởng codebase chính.
  • Bắt đầu từ Bug Fixes & Small Tasks Executing Plans
    Xây dựng trust trước. Cho AI làm task nhỏ, verify đúng, rồi mới tăng độ phức tạp. Không để AI refactor lớn ngay.
  • Sequential, Không Parallel Lúc Đầu GATE
    Dự án cũ: làm tuần tự từng task. Verify xong mỗi task trước khi làm task tiếp. Parallel dispatch (#9) chỉ dùng khi đã confident.
4 Testing (TDD Mandatory)
TDDSystematic Debug
▼
Dự án mới
  • RED-GREEN-REFACTOR TDD TDD
    Mỗi feature: viết test fail trước (RED) → code đủ để pass (GREEN) → refactor cho đẹp (REFACTOR). Không bỏ qua bước nào.
  • Edge Case Discovery
    AI generate edge cases, boundary tests, error scenarios. Coverage cao từ đầu. Không nợ test debt.
  • Debug = 4-Phase Root Cause Systematic Debug
    Khi test fail: investigate → pattern analysis → hypothesis → minimal fix. Không patch symptom, fix root cause.
Dự án cũ
  • Viết Test Cho Code Cũ TRƯỚC TDD GATE
    Quan trọng nhất! AI generate tests cho code chưa có test. Regression safety net phải có TRƯỚC khi thay đổi bất kỳ dòng code nào.
  • Systematic Debug cho Legacy Bugs Systematic Debug
    Bug cũ: AI không guess fix. 4-phase debug: hiểu root cause → viết failing test cho bug → fix → verify. 15-30 min thay vì 2-3h random fixes.
  • Human Verify Test Logic HUMAN
    Developer review test để đảm bảo test đúng business logic. AI có thể tạo tests pass nhưng không verify đúng. Đặc biệt quan trọng với code cũ.
5 Code Review
Receive ReviewRequest Review
▼
Dự án mới
  • AI Request Review Request Review
    Agent tự động request review: check plan alignment, test coverage, code quality. Severity-based: blocking vs non-blocking issues.
  • AI Receive & Fix Feedback Receive Review
    Agent đọc feedback, phân loại severity, fix blocking issues trước khi human review. Giảm thời gian review 60%+.
  • Human Final Approve HUMAN
    Senior developer review kết quả cuối cùng. AI đã xử lý 80% issues, human chỉ cần focus vào design decisions.
Dự án cũ
  • Review Trong Context Code Cũ Request Review
    Review check thêm: consistency với patterns cũ, backward compatibility, no unintended side effects trên code khác.
  • Regression Risk Assessment Receive Review
    AI list tất cả files/features có thể bị ảnh hưởng. Suggest test commands cần chạy để verify. Không chỉ review code mới.
  • Domain Expert PHẢI Review HUMAN GATE
    Bắt buộc: Người hiểu business domain review. AI không hiểu hết business rules của dự án cũ. Không được skip.
6 Verify & Finalize
VerificationBranch Finalize
▼
Dự án mới
  • Verification Checklist Verification GATE
    AI verify: code match plan? tests all pass? coverage met? no TODOs left? no placeholders? Evidence-based, không tự claim "done".
  • Branch Finalize & Merge Branch Finalize
    AI clean up branch, squash commits nếu cần, tạo PR summary. Merge strategy: merge commit hoặc rebase tùy team convention.
Dự án cũ
  • Extended Verification Verification GATE
    Verify thêm: regression tests pass? Không break existing features? Performance không giảm? Backward compatible?
  • Gradual Merge Branch Finalize HUMAN
    Deploy từng batch nhỏ. Monitor sau mỗi merge. Feature flags nếu cần. Không merge tất cả AI changes cùng lúc.
  • Rollback Plan PHẢI Có GATE
    Mỗi merge phải có rollback plan. Nếu something breaks, revert trong 5 phút. Không chấp nhận "chắc không sao đâu".

Nguyên Tắc Cốt Lõi Của Superpowers
Design Trước, Code Sau
KHÔNG viết code khi chưa có design doc được human approve. Hard gate, không exception.
TDD Là Bắt Buộc
Mỗi task: viết test fail trước, implement, verify. Không có "viết test sau" hay "skip test vì gấp".
Evidence, Không Tự Claim
Verification (#3): AI phải CHỨNG MINH work done bằng evidence (test pass, coverage report), không được nói "đã xong".
Human Là Owner
AI là force multiplier, không phải replacement. Developer hiểu 100% code, approve mọi decision. Trách nhiệm thuộc developer.
